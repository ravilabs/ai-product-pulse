# AI Product Pulse

Deterministic triage for AI product features, built around a three-layer framework: Model Performance, Product Behaviour, Business Outcome. Ships as an agent skill for Claude Code, Cursor, OpenClaw, and Hermes, and follows the open [agentskills.io](https://agentskills.io) standard.

## What it does

Most AI product teams can describe what they're tracking. Few can say, without checking, whether that coverage is real or accidental — whether the layer that actually funded the feature has a metric attached to it at all. AI Product Pulse takes whatever evidence a team has about an AI feature, scores each of the three layers against a fixed maturity ladder, and returns a report: an overall score, a maturity level, and a set of verdicts naming the specific gap. Business-Outcome Orphaned. Trust Failure Signal. Vanity-Metric Risk.

The scoring math is deterministic. The one judgment call — reading the evidence and assigning a 1-to-5 score per layer — belongs to whichever agent is running the skill, not to an API call this project owns.

## Quickstart

```bash
git clone https://github.com/ravilabs/ai-product-pulse.git
cd ai-product-pulse
pip install -e packages/python

./install-skill.sh claude-code
```

Then add the MCP server to your `.mcp.json` — see `mcp/claude-code-mcp.json.example` — and ask your agent to triage an AI feature. It'll find the skill and the two tools it needs: `triage` for a single feature, `aggregate_product_pulse` for rolling several features up into one product-level pulse.

Cursor, OpenClaw, and Hermes work the same way:

```bash
./install-skill.sh cursor
./install-skill.sh openclaw --global
./install-skill.sh hermes --global
```

No MCP support in your harness? The CLI mirrors both tools exactly:

```bash
ai-product-pulse triage --input evidence.json
ai-product-pulse aggregate --input features.json
```

`examples/sample-input.json` and `examples/sample-report.json` show a real input and its real output — generated by actually running the code, not hand-typed, so they're guaranteed to match the current schema.

## The three layers

| Layer | Question it answers |
|---|---|
| Model Performance | Is the AI doing what it was built to do? |
| Product Behaviour | Are users interacting with AI output the way it was designed for? |
| Business Outcome | Is the AI feature moving the metric it was funded to move? |

Each layer carries four indicators — see `framework.json` for the full definitions, thresholds, and the maturity ladder every score is judged against.

## Reading a verdict

| Verdict | What it means |
|---|---|
| Balanced | Every layer clears Instrumented. A floor, not a ceiling. |
| Business-Outcome Orphaned | No business metric defined at all. The single most common and most expensive gap. |
| Trust Failure Signal | The model measures well internally, but the override rate predicts abandonment before it shows up as churn. |
| Vanity-Metric Risk | Business Outcome is tracked, but shallowly, next to layers that are heavily instrumented. A metric exists; it's decorative. |
| Blind Spot: [layer] | Model Performance or Product Behaviour went untracked while the team clearly had the capability to instrument, evidenced by another layer being solid. |

A feature can carry more than one verdict at once. That's not a conflict — it means more than one thing is true about it.

## How it's built

The framework itself — layers, indicators, verdict rules, the maturity ladder — lives in `framework.json` as data, not scattered across code. Everything derived from it (`framework.yaml` for readability, `report.schema.json` for the output contract, `docs/mcp.md` for the live tool reference, the copy bundled inside the Python package) is generated, never hand-edited; a CI workflow regenerates all four on every push and fails the build if any of them drift from what's committed.

Full design rationale, with citations: [`docs/architecture.md`](docs/architecture.md). Honest limits, including the one part of this pipeline that isn't deterministic: [`docs/failure-modes.md`](docs/failure-modes.md).

The code itself follows Hexagonal Architecture (Cockburn, 2005): a domain core that knows nothing about how it's called, wrapped by ports and adapters for the two ways an agent actually reaches it — an MCP server and a CLI, both calling the identical underlying functions. Tool discovery follows the Model Context Protocol's own native primitives. Neither of these is a framework this project invented; both are cited, checkable, and older than this repository.

Python is the reference implementation. A Node/TS port is planned, parity-tested against it rather than developed independently.
